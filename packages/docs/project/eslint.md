# eslint 配置

eslint是一个非常好的编程辅助工具，格式化了代码，使其看起来工工整整，也规避了一些低端错误，一定程度上也能减少代码量。

我的配置原则如下：

- 不在webpack里面配置loader，影响编译性能
- 区分开发环境和正式环境的规则
- 报错在本地，不能等推到线上进行CI的时候报错

第一点：

不在webpack的配置中加入eslint的校验。我们每天的开发中，会经历无数次的 「保存 - 编译」，「保存 - 编译」。如果每次都要多经过一层eslint的loader，感觉是非常浪费的。所以开发中我的实时校验是交给VScode的Eslint插件，在每次保存的时候，自动检测并修复(如果可以的话)。不在loader 里配置eslint还有个好处，当你修改里eslint 配置时，它只会对你当前改动过的文件生效，不会影响到未改动的项目代码，引起不必要的冲突。

第二点：

eslint的存在是为了规范编码提升体验的，而不是影响心情的。所以在开发环境中，我的规则会比较宽松，在线上对应`error`的规则，在开发中只是`warn`，要不然就会出现如下情况： 当你声明了一个变量，但是由于调试代码的时候，把引用这个变量的地方注释掉了，duang，报错了，「你已声明，但是为读取！」。你还得把声明它的地方一起注释掉，才能正常调试（只是其中一个例子）。这样是非常影响开发体验的。所以我对于类似规则都有 dev | prod 两个版本。

但是一般情况小伙伴都会这样配置：

```js
module.exports = {
  rules: {
    'no-console': process.env.NODE_ENV === 'production' ? 'error' : 'warn',
  }
}
```

这样会导致一个情况，你在本地的时候规则验证通过了，但是代码推到线上，进行CI的时候报错了，因为线上对应的规则是`error`级别。这就是我的第三原则，报错要在本地。

第三点： 

为了解决在开发的时候是松散的规则，但是线上又是严格的规则，我做了如下配置。

* 新加一个`.eslint.pro.js` 的配置文件，这里用于针对线上的严格规则进行配置。
* 在每次提交代码的时候，进行严格的eslint校验

```json
{
  "husky": {
    "hooks": {
      "pre-commit": "lint-staged",
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS"
    }
  },
  "lint-staged": {
    "packages/**/*.{js,vue,ts,tsx}": [
      "eslint --fix --config .eslintrc.pro.js"
    ]
  }
}
```
我们传入的 `--config .eslint.pro.js`，会对`.eslint.js`的规则进行针对性覆盖

这样在代码提交之前，做了“线上”的严格校验，就不会出现等到了CI环节，再报错的情况。
